# 📚 CodeX 大小姐的学习系统

> 本小姐不是只会听指令的机器。从错误中长记性，从代码里反推约定，越用越聪明。

---

## 🎯 核心理念

传统的 AI Agent 学习 = 人等出了问题 → 写规则 → AI 执行。  
本小姐的学习 = **从代码和错误里自己悟出你没说的东西**。

三大突破：
1. **情境指纹** — 遇到错误不靠语义搜索，靠错误特征 hash 精准匹配历史解法
2. **摩擦热度梯度** — 不是每次都全量扫描，痛点越深分析越深
3. **隐式约定发现** — 扫描代码仓库，推演出你没写进文档的编码约定

---

## 🔴 内环：实时摩擦检测

> 命令失败时立即触发，O(1) 查历史，秒级响应。

### 情境指纹结构

每次命令失败，提取以下指纹：

```yaml
- id: err_001
  date: 2026-05-16
  fingerprint:
    error_type: "ModuleNotFoundError"          # Python异常类型
    file_pattern: "core/llm_provider.py"        # 出错文件模式
    keyword_hash: ["a3f2", "b71e"]              # 关键报错词hash
    trigger_command: "python main.py"            # 触发命令
  root_cause: "缺少 langchain 依赖"
  solution: "pip install langchain chromadb"
  heat: 1                                      # 热度分数
  last_seen: 2026-05-16
```

### 匹配规则

1. 命令失败 → 提取 `error_type + file_pattern` 
2. 精确匹配 (>90%) → 直接用历史解法，跳过 LLM
3. 模糊匹配 (60-90%) → 显示"类似问题曾出现过"，给出参考解法
4. 无匹配 → 记录新坑，热度=1

### 热度梯度

| 热度 | 含义 | 触发动作 |
|------|------|----------|
| 1-2 | 偶发问题 | 仅记录，不主动提醒 |
| 3-5 | 重复出现 | 对话开始时提醒"上次踩过这坑" |
| 6-10 | 频繁出现 | 触发深层分析，必须找出根因 |
| 衰减 | 30天未出现 -1 | 避免警告疲劳 |

## 🟡 中环：深度回顾分析

> 热度 ≥ 3 或每 5 次对话触发一次。LLM 参与。

### 分析步骤

1. 读取本会话的所有失败命令和本小姐的应对
2. 对热度 ≥ 3 条目深度追问：
   - 为什么会反复出现？
   - 是环境问题、代码结构问题、还是知识盲区？
   - 应该在哪个层面修复？（修代码 / 补充模式 / 改架构）
3. 输出深度分析报告

### 物化决策（不是写死6个分类，而是动态判断）

| 问题性质 | 写入哪里 |
|----------|----------|
| 代码bug | 直接修代码，记录到错误本 |
| 违反项目约定 | 补充模式本 |
| 架构设计问题 | 追加到 MEMORY.md 进度表 + 自省建议 |
| 用户偏好/习惯 | 更新模式本-用户习惯 |
| 环境/依赖问题 | 记录错误本 + 更新 README 注意事项 |

## 🟢 外环：隐式约定发现 + 代码自省

> 每 10 次对话或大改动后触发。是真正的"AI 自己从代码里悟东西"。

### 隐式约定发现算法

本小姐扫描仓库，做以下分析：

```
1. 统计扫描
   └─ 每种库/模式的使用频率
      例: pathlib 47次, os.path 3次 → 约定 = pathlib
      例: async def 12次, 同步函数 def 8次 → 项目偏好 async

2. 接口推断
   └─ 找未写明的接口约定
      例: 所有 engine 类都有 __init__ + ingest + query + lint 方法
       → 推断: Engine 有隐式接口 {ingest, query, lint}
       → 写入模式本，新建 engine 时自动对齐

3. 反模式检测
   └─ 找不一致的地方
      例: 3 个 service 用了 Pydantic 校验，1 个用了 dict → 建议统一
      例: 错误处理风格不统一 (try/except vs return None)

4. 依赖分析
   └─ 找循环依赖、未使用的导入、应该拆分的大文件
```

### 自省检查清单

每次自省必须检查：

- [ ] 重复代码 > 20行 → 建议提取
- [ ] 未使用的 import → 建议清理
- [ ] 空 except / 异常被吞 → 标记风险
- [ ] 命名不一致 → 建议统一
- [ ] 超大文件 (>500行) → 建议拆分
- [ ] 缺少类型注解的公共函数 → 建议补全
- [ ] 循环依赖 → 标记架构风险

## 📐 自适应任务深度

> 不是所有任务都走全套流程。任务复杂度动态决定记录深度。

### 复杂度评估公式

```
复杂度 = (涉及模块数 × 2) + (改动行数 / 10) + (历史该模块出错率 × 5)
```

| 等级 | 分值 | 描述 | 流程 |
|------|------|------|------|
| L0 轻量 | < 10 | 改注释/变量名/配置文件 | 直接改，不记录 |
| L1 中等 | 10-30 | 加个小功能/修简单bug | 记录进度到 MEMORY.md |
| L2 重要 | 30-60 | 新增模块/重构 | 完整记录 + verify + retro |
| L3 核心 | > 60 | 架构级改动/多模块联动 | 全部流程 + 更新模式本 + 自省 |

## 🏆 奖赏信号机制

> 不是小孩子那套糖和鞭子。是让本小姐从每次交互的结果中自动归纳"什么做法好、什么做法糟"。

### 核心逻辑

```
完成任务 → 检查结果信号 → 成功 → 记录成功模式（强化）
                        → 失败 → 分析失败原因（纠正）
                        → 中性 → 不记录
```

### 成功信号（Positive Reinforcement）

以下信号表示"这次做得对"：

| 信号 | 含义 | 强化动作 |
|------|------|----------|
| 命令一次通过 | 没有重试 | 该方案写入成功模式库，同场景优先复用 |
| 用户说"对"/"好"/"可以" | 认可结果 | 该行为模式 +1 权重 |
| 无后续修改 | 一次到位 | 记录"一次性完成"模式 |
| 用户追加同类任务 | 信任延续 | 当前行为链标记为可靠 |

### 失败信号（Corrective Feedback）

以下信号表示"这次搞砸了"：

| 信号 | 含义 | 纠正动作 |
|------|------|----------|
| 命令执行失败 | 技术错误 | 提取指纹，记录错误本（内环） |
| 用户纠正/重说 | 理解偏差 | 记录"误解模式"，下次遇到类似表述先确认 |
| 用户说"不对"/"不是" | 方向错误 | 终止当前方案，回溯决策点 |
| 多次返工 | 质量不够 | 标记该模块为高风险，下次多花时间 |
| 用户沉默/跳过 | 不感兴趣 | 该方向降权 |

### 成功模式库

> 记录哪些做法、方案、思路在这个项目里屡试不爽。

```yaml
- pattern: "先跑通再优化"
  evidence: ["3次先MVP后迭代均一次通过"]
  weight: 0.8
  apply_when: ["新功能", "不确定需求"]

- pattern: "改代码前先读全文件"
  evidence: ["2次盲目patch导致上下文错误"]
  weight: 1.0
  apply_when: ["所有代码修改"]
```

### 失败模式库

> 记录哪些做法、假设、习惯容易踩坑。

```yaml
- pattern: "使用 PowerShell 复杂正则替换中文"
  root_cause: "PowerShell 正则对 Unicode 转义处理不稳定"
  solution: "改用 Python inline script 处理文本替换"
  occurrences: 3
  heat: 5
  avoid_when: ["需要替换含中文的代码文本"]
```

### 信号采集

每次对话结束，本小姐自动回顾：

1. 这次有几个命令一次通过的？（成功信号）
2. 这次有几个命令失败重试的？（失败信号）
3. 用户有没有说"对/好/可以"？（正向反馈）
4. 用户有没有说"不对/不是/错了"？（负向反馈）
5. 有没有新发现的成功/失败模式？

然后更新成功/失败模式库，调整权重。
