## 核心模块详解

### 1. LLM Provider（大模型调用层）

支持多种 LLM 后端，通过工厂模式统一调用接口：

| 提供商 | 配置字段 | 默认模型 |
|--------|---------|---------|
| OpenAI | `openai_api_key` | `gpt-4o` |
| DeepSeek | `openai_api_key` + `openai_base_url` | `deepseek-chat` |
| Ollama | `ollama_base_url` | `qwen3` |
| 智谱 | `zhipu_api_key` | `glm-4` |

当前系统实际配置为 **DeepSeek** (`deepseek-chat`)，通过 OpenAI 兼容接口调用。

### 2. QueryService（查询服务）

查询服务是整个系统的核心编排层，支持三种工作模式：

```
query_with_mode(question, mode, top_k)
  │
  ├── mode="pipeline" → self.query() → Wiki 优先 → RAG 兜底 → 知识复利
  │
  ├── mode="agent" → ReActAgent.run() → Agent 自主决策
  │
  └── mode="auto" → 根据问题特征自动选择
       ├── 短问题/简单问题 → pipeline
       └── 长问题/复杂问题 → agent
```

### 3. 缓存系统

- **查询缓存**：基于 MD5 哈希，TTL 1 小时
- **评分缓存**：答案质量评分缓存，TTL 24 小时
- **会话历史**：保留最近 10 轮对话上下文

## 数据流示例

以下是一个典型的知识问答完整数据流：

```
1. 用户在聊天界面输入问题
2. POST /api/query → QueryService.query_with_mode()
3. 检查缓存（命中则直接返回）
4. Wiki 引擎查询 → 如果命中直接返回
5. RAG 引擎兜底：
   a. Query 改写（补充关键词）
   b. Vector 检索（ChromaDB）
   c. LLM 生成回答
6. 答案质量评分
7. 高分答案自动沉淀到 Wiki（知识复利）
8. 返回结果给前端
```

## 设计决策

1. **为什么使用双引擎？** Wiki 适合精确匹配和结构化知识，RAG 适合模糊语义搜索，两者互补
2. **为什么选 ChromaDB？** 轻量级、嵌入式、无需单独部署，适合中小企业场景
3. **为什么答案需要评分？** 确保只有高质量回答才沉淀到 Wiki，避免知识污染
