# RAG 引擎

## 概述

RAG（Retrieval-Augmented Generation）引擎是 CodeXFiles 的语义检索核心，负责将用户问题转换为向量并在文档库中检索最相关的内容。当 [[系统架构]] 中的 Wiki 引擎无法精确命中时，RAG 引擎作为兜底方案确保用户始终能获得回答。

## 核心流程

```
用户问题
   │
   ▼
┌──────────────┐
│ Query 改写    │ ← 补充同义词、术语，提升召回率
└──────┬───────┘
       ▼
┌──────────────┐
│ Embedding 向量化│ ← SentenceTransformer
└──────┬───────┘
       ▼
┌──────────────┐
│ ChromaDB 检索  │ ← 余弦相似度 + BM25 混合检索
└──────┬───────┘
       ▼
┌──────────────┐
│ 重排序 (Rerank)│ ← 取 Top-K 重新排序
└──────┬───────┘
       ▼
┌──────────────┐
│ LLM 生成回答   │ ← 结合上下文 + 检索结果
└──────────────┘
```

## Embedding 模型

系统使用 **BAAI/bge-small-zh** 作为默认 Embedding 模型：

| 属性 | 值 |
|------|-----|
| 模型名称 | BAAI/bge-small-zh |
| 向量维度 | 512 |
| 语言 | 中文优化 |
| 设备 | CPU（默认） |
| 加载方式 | SentenceTransformer |

bge-small-zh 是北京智源人工智能研究院（BAAI）推出的面向中文的轻量级 Embedding 模型，在中文语义检索任务上表现优异，且推理速度快、资源占用低。

## 文档分块策略

文档摄入时按以下规则进行分块（Chunking）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| chunk_size | 500 | 每个块的最大字符数 |
| chunk_overlap | 50 | 相邻块的重叠字符数 |
| 分块方式 | RecursiveCharacterTextSplitter | 递归分割，优先保持段落完整 |

## 检索策略

### 混合检索
RAG 引擎支持两种检索策略的组合：

1. **向量检索（语义搜索）**
   - 基于 Cosine Similarity 计算语义相似度
   - 擅长捕捉同义词和语义相近的内容

2. **BM25 检索（关键词搜索）**
   - 基于关键词精确匹配
   - 擅长捕捉专有名词和精确术语

```
最终得分 = 0.7 × 向量相似度 + 0.3 × BM25 得分
```

### 检索参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| retrieval_top_k | 10 | 检索返回的候选文档数 |
| rerank_top_k | 3 | 重排序后最终使用的文档数 |
| similarity_threshold | 0.5 | 相似度阈值，低于此值的文档被过滤 |

## 答案质量评估

RAG 引擎每次生成回答后，会调用 LLM 对答案进行评分（1-10 分）：

- **≥ 8 分**：高质量答案，自动沉淀到 Wiki（知识复利）
- **5-7 分**：中等质量，直接返回
- **< 5 分**：低质量，标记为低置信度

评分标准：
1. 是否完整回答了问题（0-4 分）
2. 是否有充分的资料来源支撑（0-3 分）
3. 是否有编造或推测的内容（有则扣 0-3 分）

## 知识复利机制

当答案评分 ≥ 7 分时，系统自动将问答对转换为 Wiki 页面存入 [[系统架构]] 的 Wiki 引擎中，实现以下效果：

- **一次问答，永久受益**：高质量的知识自动沉淀
- **Wiki 持续丰富**：不需要人工手动录入
- **冷启动问题解决**：初期知识库为空时，通过 RAG 回答逐步积累

## ChromaDB 向量数据库

| 属性 | 值 |
|------|-----|
| 数据库类型 | ChromaDB（嵌入式） |
| 持久化目录 | ./chroma_db |
| 集合名称 | enterprise_knowledge |
| 部署方式 | 无需单独部署，与应用同进程 |

ChromaDB 是一个轻量级的向量数据库，所有数据持久化在本地磁盘，无需额外的数据库服务。
